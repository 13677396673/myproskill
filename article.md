# 让 Agent 帮你做"看起来像视频"的网页演示：web-video-presentation Skill 设计实录

## 前言

如果你试过让 AI 帮你做演示视频，大概遇到过这些尴尬：

- 生成的是传统幻灯片 —— 逐页翻，像 PPT，不像视频
- 把文章转成视频脚本 —— 但画面全是纯文字，录出来没人想看
- 想要一点"电影感" —— 结果得到紫粉渐变背景 + 圆角彩色卡片 + emoji 当图标

这是我在开发 Claude Code Skill `web-video-presentation` 时试图解决的核心问题：**如何把一篇文章或口播稿，变成一段"看起来像视频"的点击驱动网页演示**，而且输出物要真的能看、能录、能发出去。

---

## 它解决什么问题

这个 Skill 的输入是一篇文章、一篇口播稿，或者一个"帮我做个 X 主题视频"的需求。输出是一个 Vite + React + TypeScript 项目，16:9 固定画幅，每次点击推进口播稿的一个节拍，每一步独占整屏。

如果你想把网页录成视频，它还提供一套 provider-agnostic 的音频合成系统，能自动合成每 step 一句口播。合成后用 `?auto=1` 模式打开页面，按 Space 就能自动跑完全片 —— 打开屏幕录制，一镜到底，音画天然同步，**不需要后期对轨**。

适用场景很具体：

- 把公众号文章转成 B 站 / YouTube / 视频号口播视频
- 做产品演示、教程、视觉 talk，"动态 PPT 但不像 PPT"
- 做有电影感的路演 demo

---

## 核心设计哲学

这个 Skill 沉淀的不是一个模板，而是一套**方法论 + 协作流程**。脚手架只提供舞台原语和主题 token，每个项目的美学决策（配色、字型、动效气质）都应该针对内容重新设计。

### 十条原则

我从大量实践中浓缩了十条原则，写在 `CHAPTER-CRAFT.md` 里。挑几条我最想展开讲的：

**固定 16:9 舞台**：内容在 1920×1080 坐标系里设计，外层用 `transform: scale` 缩放到任何视口。没有响应式断点，没有流式布局 —— 这就是视频，不是网页。

**每步独占整屏**：章节是 step 的纯函数，没有定时器，没有命令式状态。`if (step === N) return <FullScene />`，简单到不可能出错。

**内容驱动动画，不是入场动画兜底**：大部分 AI 生成的演示，动画是一套淡入效果反复用。这个 Skill 要求每步先找"内在动作"—— 数字递增、排名交换、流程节点依次点亮、对比一刀切开。实在找不到才用入场动画。这一步区分了"视频"和"翻 PPT"。

**口播节拍 = step**：口播稿里用 `---` 切分一个完整的聚焦想法，outline 里的一个 step 就对应这一段。视频节奏不是设计师拍脑袋定的，是口播稿自然决定的。

**隐藏 chrome**：进度条和翻页控件默认 `opacity: 0`，鼠标悬浮到底部边缘才出现。录屏时画面没有任何浏览器痕迹，看起来就是一段成品视频。

### 双源原则

这是设计上最有意思的一条。每一章有**两个信息源**：

- **口播稿（script.md）**决定节奏和顺序，一句话对应一个 step
- **原文（article.md）**决定画面信息密度

实现章节时，口播稿决定讲到哪了，但画面上的细节要从原文里挖 —— 具体数字、引用原话、案例维度、出处时间。让"画面信息密度 > 口播信息密度"，屏幕不仅仅是把口播打字打了一遍。

> 如果你只用了口播稿的内容做章节，屏幕等于把口播打字打了一遍 —— 那就是 PPT，不是视频。

## 工作流：从文章到视频

整个流程被设计成**硬 Checkpoint 分段协作**，而不是闷头跑到底。

### Phase 1：一次产出两份文件

输入一篇原文，Agent 在一次思考里产出两份东西：

1. **script.md**：口语化口播稿。遵循 SCRIPT-STYLE.md 的 8 条原则（短句 ≤ 20 字、第二人称、3 秒钩子、数字翻译成感受……）和"去 AI 味五类"（假共情、假深刻、自我标榜、万能模板、排比堆砌）
2. **outline.md**：开发计划。只规划节奏与信息密度 —— 章节切分、每步屏幕内容、章节级信息池。**不规划动画**（动画由章节开发时按内容自由设计）

两份产出物都要走**硬性自检协议**，优先用独立的 reviewer agent 逐项核查，改完才能进入下一阶段。

### Checkpoint Plan：5 件事一次对齐

这是第一个硬节点。让用户一次确认：

1. 稿子要不要改
2. 开发计划（outline）要不要改
3. 选哪个主题（从 23 套内置主题中推荐 2~3 套最匹配的）
4. 素材怎么准备
5. 开发模式（逐章确认 / 顺序开发 / 并行开发）

主题选择会影响动效气质，outline 确认能避免章节节奏跑偏。**这些 Checkpoint 是 Skill 契约的一部分** —— Agent 不应该从原文一路闷头做到成品。

### Phase 2：网页开发

脚手架用 `scaffold.sh` 一键生成项目，按选定的主题注入 token。

**第 1 章必须在主线程做，且必须做到完整版本 + 用户验收** —— 这是强制 anchor。第 1 章是 CHAPTER-CRAFT 指引在"当前主题 + 当前题材"下的第一次落地，如果有盲区，早改成本最低。

第 2 章及之后支持三种模式：

- A（默认）逐章确认 —— 每章做完暂停验收，风险可控
- B 顺序开发 —— 第 2~N 章主线程顺序做完，统一验收
- C 并行开发 —— 用 subagent 并行做，速度最快，风格差异是设计预期

## 主题系统：23 套独立设计 DNA

我花了很大精力做主题系统。每个主题不是简单地换颜色，而是设计了**独立的设计签名**：

- `midnight-press`（深色印刷）：暖色 espresso 底 + 火热橙，Instrument Serif italic 英文，慢速电影感节奏。只有 vignette，没有颗粒
- `newsroom`（报社）：报纸奶油 + 墨黑衬线 + 旗红，0 圆角（报纸不会圆角）。安静的印刷节奏
- `terminal-green`（终端绿）：纯黑 + JetBrains Mono only + CRT 扫描线 + 磷光 text-shadow。利落线性动效
- `chalk-garden`（粉笔花园）：深石板黑板 + Patrick Hand 手写体 + 粉笔黄 + 2px 虚线 rule + film grain
- `bauhaus-bold`（包豪斯）：0 圆角 + 4px 实色厚边 + 4px 黑色画框包住舞台
- `split-canvas`（双拼画布）：屏幕本身就是蜜桃 + 薰衣草 50/50 双色硬切分
- ……

共 23 套深色和浅色主题，每套 25~35 个 CSS token（颜色、字体家族、圆角性格、分割线性格、hero 数字风格、舞台 padding 密度、装饰层）。

主题切换是一次 `cp` 命令的事。刷新 dev server 就能看到新面貌，章节代码一行不动 —— **前提是章节代码用了语义 token，而不是硬编码 hex 或字体名**。这个约束保障了"换主题不破"的契约。

## 反 AI 味：设计方法论中最硬的一刀

AI 生成的网页有几种共通的"视觉指纹"，我在 CHAPTER-CRAFT 里明令禁止：

- 紫粉 / 蓝紫对角渐变背景
- 圆角卡片 + 彩色左边框装饰
- 渐变按钮 + 大圆角药丸
- emoji 当图标
- 假数据 / 假 logo / 假"X 万用户"
- 整章 N 步同一种入场动画
- 每屏右下角挂 mono 角标

缺素材就承认缺 —— 用 placeholder 占位卡按真实比例留位。**不要用 emoji 凑、不要找无关图凑、不要编数字。没有就承认没有，比 fake 强一百倍。**

口播稿层面也有完备的"去 AI 味"规范：假共情（"我知道你最近肯定很纠结"）、假深刻（"恰恰相反"）、自我标榜（"我必须要说一句"）、万能模板（"说白了 / 底层逻辑 / 一句话总结"）、排比堆砌 —— 五类腔调必须清干净。唯一判断标准：**这话念出来，一个真人会这么说吗？**

## 技术架构

项目是标准 Vite + React + TypeScript，产出物结构：

```
my-video/
├── script.md            # 口播稿
├── outline.md           # 开发计划
└── presentation/        # Vite + React + TS 项目
    ├── src/chapters/    # 每章独立文件夹
    ├── scripts/         # 工具脚本
    └── public/audio/    # 合成音频
```

几处设计上我觉得对别人有参考价值的设计：

### step 游标模型

全局 `useStepper` hook 管理 `(chapter, step)` 游标，本地持久化到 localStorage。章节是 step 的纯函数 —— 没有组件内部状态，没有 `useEffect` 驱动动画，动画全部用 CSS keyframes 声明式完成。

每次点击舞台空白处 → `next()` → 新 step 的组件重新渲染 → CSS transition 自动触发。

这个模型约束了复杂度：**5 处地方（script / outline / 章节代码 / chapters.ts / 音频文件）永远不会漂**，因为他们的源头都是同一个 `narrations.ts`。

### Auto 模式

合成音频后，`?auto=1` 模式可以实现全自动播放。运行规则就一句话：每段音频播完 + 200ms 缓冲 → 自动推进下一步。

没有"等动画跑完"的兜底。如果你的视觉动画比口播长，会被当场切断。解决办法很直接：写更长口播 / 拆 step / 调动画速度。**这个设计迫使动画时长必须服务于口播节奏，而不是反过来**。

### Provider-agnostic 音频

合成器不绑定任何 TTS 后端。每个后端是 `scripts/tts-providers/<name>.sh` 一个文件，暴露 `tts_synthesize(text, out_path, voice)` 等三函数契约。内置 minimax、OpenAI TTS、小米 MiMo-V2.5-TTS、MiMo VoiceClone 四个 provider。README 里还附了 5 段现成代码片段（ElevenLabs / edge-tts 免费 / macOS say 离线 / Azure / Google Cloud）。

## 硬性自检协议：防止 Agent 偷懒

做 Agent Skill 最怕的是 Agent 为了省事跳过质量步骤。这个 Skill 在每个关键产出点都设置了**硬性自检流程**：

1. `script.md` 写完 → SCRIPT-STYLE 三层自检（形式 / 风骨 / 念出来）
2. `outline.md` 写完 → OUTLINE-FORMAT 自检
3. 每章实现完成 → CHAPTER-CRAFT 完工自检（20+ 项：视觉演示、逐步揭示、反 AI 味、token 正确性、TypeScript 类型检查、narrations 对齐）

执行方式按能力降级：Agent Teams → subAgent → 自检。**拿结论汇报但不修 = 违规**。

这套自检协议比具体代码规则更重要 —— 它是"让 Agent 自己给自己把关"的机制设计，脱离了这个问题，任何 Skill 都是单向输出。

## 尾声：方法论比模板重要

这个 Skill 从第一版到现在，迭代最多的地方不是代码，而是**协作流程的设计**。什么阶段该停、该问用户什么、怎么防止 Agent 自作主张、怎么确保产出物的一致性 —— 这些比技术选型更影响最终质量。

如果你也在做 Agent Skill，我最想分享的建议是：

1. **Checkpoint 设计比提示词重要**。什么时候停、怎么对齐、谁决策 —— 决定了 Agent 是工具还是甩手掌柜
2. **先定原则，再写代码**。十条原则在 CHAPTER-CRAFT 里，所有开发决策从原则往回推理，而不是从模板出发
3. **去 AI 味不是降质**。把"AI 在朗诵"换成"人在聊天"，信息量一个不少，腔调全部换掉。如果换了之后内容变蠢 —— 它本来就蠢，是该补内容了，不是去 AI 味的锅
4. **主题系统换的是签名，不是颜色**。23 套主题每套有自己的设计语言（线条性格 / 圆角哲学 / 装饰签名 / 动效基线），不是配色彩板

如果你感兴趣：完整代码在 [GitHub](https://github.com/ConardLi/garden-skills/tree/main/skills/web-video-presentation)，文档是中英双语的，Skill 本身是 Claude Code 的 plugin，也兼容 Cursor、Codex CLI、Gemini CLI、OpenCode 等 Agent 工具。
